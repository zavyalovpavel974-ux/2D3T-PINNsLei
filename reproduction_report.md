# 2D 3T PINNs Reproduction Report

## Current Status

- Author code is present in this repository.
- Public author repository did not include the required `sol1_...txt` numerical
  reference files.
- A reference-generation workflow has been added under `reference_solver/`.
- The SciPy reference solver now uses an RMS Newton residual criterion so the
  nonlinear tolerance is not tied to the number of grid points.
- Previous-worker FBPINN artifacts include valid low-resolution `20x20`
  snapshots, but those are not paper-level `80x80` references.
- The previous `reference_example5.npz` from `Desktop/code reproduction` is not
  usable because all three reference fields are NaN.
- Strict `20x20` references have been generated for both target cases and
  exported in the author text format.
- `80x80_from20` files have been generated by interpolation only for author-code
  smoke tests. They are not strict paper reference data.
- A short inverse-script smoke test with those interpolated files passed data
  loading and entered GPU training; it was cut off by the 120 s timeout.
- A short Example 5 smoke test created its output directory and entered the
  first `Aei=70` training stage; it was also cut off by the 120 s timeout.

## Commands

Strict low-resolution reference generation:

```powershell
python reference_solver/generate_reference.py solve --case aei70_krar --nx 20 --ny 20 --out reference_exports\aei70_krar_20\reference_snapshots.npz --dt-init 5e-3 --dt-max 2e-2 --newton-max 8 --gmres-maxiter 120
python reference_solver/generate_reference.py solve --case aei700_krartr --nx 20 --ny 20 --out reference_exports\aei700_krartr_20\reference_snapshots.npz --dt-init 5e-3 --dt-max 2e-2 --newton-max 8 --gmres-maxiter 120
python reference_solver/generate_reference.py validate --npz reference_exports\aei70_krar_20\reference_snapshots.npz --json-out reference_exports\aei70_krar_20\validation.json
python reference_solver/generate_reference.py validate --npz reference_exports\aei700_krartr_20\reference_snapshots.npz --json-out reference_exports\aei700_krartr_20\validation.json
python reference_solver/generate_reference.py export --case aei70_krar --npz reference_exports\aei70_krar_20\reference_snapshots.npz --out-dir reference_exports\aei70_krar_20
python reference_solver/generate_reference.py export --case aei700_krartr --npz reference_exports\aei700_krartr_20\reference_snapshots.npz --out-dir reference_exports\aei700_krartr_20
```

## Open Risks

- The public forward script appears to read only the `t=1` Example 5 reference
  text file for all diagnostic blocks, despite the paper discussing multiple
  time snapshots.
- Full training remains hour-scale.
- The current SciPy/JFNK reference solver is not fast or robust enough for
  direct strict `80x80` production.
- Strict agreement with paper tables depends on obtaining an `80x80` reference
  solver closer to the authors' unpublished finite-volume/KINSOL implementation,
  or accelerating the current solver substantially.
